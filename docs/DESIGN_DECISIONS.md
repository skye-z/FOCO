# 设计取舍

## 1. Supabase + PostgreSQL 作为唯一事实源

理由：

- 题库、练习、画像、交互单元都需要强一致写入。
- Supabase 自带 Auth，和前端登录链路对齐。
- PostgreSQL 的 JSONB 适合保存题目、步骤、反馈和推荐证据。

## 2. Go + xorm + 原生 SQL

理由：

- 现有代码已经以 xorm 为主，不需要再引入第二套 ORM。
- 高频学习聚合本来就需要原生 SQL。
- 版本化内容和练习快照更适合显式 SQL。

## 3. 内容采用版本化而不是原地覆盖

理由：

- 已发布题目 / 交互单元必须可追溯。
- 历史练习需要保留当时看到的题干、选项、解析。
- 编辑时只克隆新版本，避免污染历史数据。

## 4. session 化练习

理由：

- 刷新页面、网络重试、重复提交都不应该改变题目集合。
- 练习结果必须与题目快照一一对应。
- `practice_sessions` / `practice_session_items` 解决了状态保留。

## 5. 学习智能化优先做可解释规则

理由：

- 备考场景里，用户更关心“为什么推荐我做这个”。
- 先把规则链路跑稳，再考虑更复杂的模型或 AI。
- 当前的推荐理由已经可以直接解释给用户看。

## 6. 学习端 / 管理端共用同一套 API 包裹

理由：

- 前后端都在同一仓库，统一响应结构便于调试和测试。
- 方便 E2E 和 API collection 直接复用。
- 减少前端对异常格式的分支处理。

## 7. 缓存策略分层

现状：

- 当前代码已接入 Go 本地缓存 + Redis 二级缓存。
- `REDIS_URL` 只配置连接信息；管理页设置仍保存在 `admin_settings` 表。
- 首页、画像、推荐、题库、知识图谱、交互单元、错题本和 token 校验都通过 namespace 版本号失效。
- L1 进程内缓存只做短 TTL 命中，L2 Redis 负责多实例共享。

原因：

- 本地缓存降低同进程重复 JSON 聚合和 DB 读取。
- Redis 缓存支持多进程 / 多容器共享。
- namespace version 避免按 key 扫描删除，写入只需要 bump 版本号。

TTL 取值思路：

- 读取频繁但变更不频繁的设置和图谱：分钟级。
- 频繁变化的学习首页、推荐、画像：秒到分钟级。
- session 这种强一致快照：短 TTL，只做重复请求消峰。

后续可继续拆：

- 后台任务：画像重算、周报、路径生成。

## 8. 交互单元以 JSON schema 存储步骤

理由：

- 不同交互类型差异大，直接建宽表会变得笨重。
- JSONB 能保留 widget 配置、评价逻辑和提示策略。
- 版本化后能支撑持续迭代而不用迁移表结构。
